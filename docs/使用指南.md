# Visual Worktree：把"多仓库 + 多分支"的混乱，收进一块看板

> 一个本地桌面工具，专治"同一个需求要改好几个仓库""手里一堆 worktree 不知道哪个是哪个"的日常痛点。

---

## 一、先说说背景：我们到底在跟什么较劲

如果你也在**多仓库**的项目里干活，下面的场景应该不陌生：

**场景一：一个需求，散落在好几个仓库里。**
做 `PROJ-2001`，前端要改 `web-app`，管理端小程序要改 `admin-wxapp`，司机端也要动。于是你 `cd` 来 `cd` 去，在每个仓库里 `git worktree add` 一遍，建出来的目录散落在各处。过两天回来，你已经记不清这个需求到底碰了哪几个仓库、各自在哪个分支了。

**场景二：手里 worktree 越积越多，状态全靠脑子记。**
`git worktree list` 能列出来，但它是**按单个仓库**列的，而且只有路径和分支，看不出哪个有没提交的改动、哪个落后远程多少。你想知道"我 hotfix 那个 worktree 改完提交了没"，得一个个 `cd` 进去 `git status`。

**场景三：清理是个体力活。**
需求上线了，对应的几个 worktree 要删。手动删目录后 git 里还留着失效引用，得再 `git worktree prune`。一套下来，跨好几个仓库重复操作。

这些事单看都不难，但它们的共同点是：**信息是散的，操作是重复的，全靠人脑去聚合和记忆。** Visual Worktree 做的就一件事——**把这些散落的信息按"任务"聚合到一块看板上，把重复操作变成点几下鼠标。**

---

## 二、它的核心思路：以"任务"为单位，而不是"仓库"

这是 Visual Worktree 跟 `git worktree list`、跟各种 Git GUI 最不一样的地方。

普通工具是**仓库视角**：「`web-app` 这个仓库有哪些 worktree」。

Visual Worktree 是**任务视角**：「`PROJ-2001` 这个需求，涉及哪几个仓库、各自什么分支、什么状态」。

它依赖一个很自然的目录约定：

```
<worktree根目录>/
  ├── PROJ-2001-商品字典字段显示对齐/      ← 一个任务
  │     ├── web-app/                        ← 任务下，某个仓库的 worktree
  │     ├── admin-wxapp/
  │     └── driver-wxapp/
  ├── hotfix-PROJ-2002/                     ← 另一个任务
  │     ├── finance/
  │     └── web-app/
  └── ...
```

也就是：**`worktree根目录 / 任务名 / 项目名`**。只要你按这个结构建 worktree（用工具的"创建"功能会自动帮你建成这样），它就能自动把同一个任务下、跨多个仓库的 worktree 聚合到一组里展示。

> 它怎么做到的？它会扫描你"源项目根目录"下的每一个 git 仓库，对每个仓库执行 `git worktree list`，然后挑出那些路径落在"worktree 根目录"下的 worktree，按路径的第一段（任务名）归类。所以**它读的是 git 的真实数据，不是自己记的账**——你在命令行手动建的 worktree，它照样能扫出来。

## 三、安装

> 目前打包的是 **macOS Apple Silicon（M 系列芯片）** 版本，格式为 DMG。

### 1. 安装应用

1. 双击 `Visual Worktree-1.0.0-arm64.dmg`
2. 把 **Visual Worktree** 图标拖到 **Applications** 文件夹
3. 从"启动台"或"应用程序"里打开

### 2. 首次打开会被系统拦截（正常现象）

这个 app 没有买 Apple 开发者签名，所以 macOS 第一次会提示"无法打开，因为来自身份不明的开发者"或"已损坏"。两种解决方式：

**方式一（推荐）：** 在"应用程序"里**右键点击** Visual Worktree → 选「打开」→ 再点「打开」。之后就能正常双击了。

**方式二：** 如果方式一还提示"已损坏"，打开"终端"执行这一行（移除系统给下载文件加的隔离标记，安全，不改动 app）：

```bash
sudo xattr -rd com.apple.quarantine "/Applications/Visual Worktree.app"
```

### 3. 一个前置条件

目标机器需要装了 `git`（绝大多数开发机都有）。没有的话执行 `xcode-select --install`。工具调用的是系统的 git，自身不内置。

## 四、第一次使用：先把两个路径配对

打开后，点右上角**「设置」**，确认两个关键路径：

| 配置项 | 说明 | 默认值 |
| --- | --- | --- |
| **源项目根目录** | 你存放各个 git 仓库的目录，工具会扫描它下面所有仓库 | `~/work/projects` |
| **Worktree 根目录** | 你建 worktree 的地方，任务就建在它下面 | `~/work/worktrees` |
| **主分支名** | 用来判断"在不在主分支"，默认 `master`、`main` | `master`, `main` |
| **自动 fetch** | 开启后扫描会先 `git fetch`，能算准"落后远程几个提交"，但更慢 | 关 |

> 改完这两个路径，工具立刻就能扫出你现有的项目和 worktree。配置存在 `~/.visual-worktree/config.json`。

界面顶部有两个视图切换，**默认进「Worktree」视图**（任务视角），旁边是「项目」视图。下面分别讲。

## 五、Worktree 视图：你的日常主战场

这是工具的核心。打开就是按任务分组的折叠面板，每个任务一组。

### 看什么

展开一个任务（比如 `PROJ-2001`），你能一眼看到：

- 这个任务涉及**哪几个仓库**（每行一个）
- 每个仓库的 worktree 在**哪个分支**
- 每个 worktree 的**实时状态标签**：
  - 🔴 **有变更 N** —— 有 N 个未提交的文件改动
  - 🔵 **领先 N** —— 本地比远程多 N 个提交（还没 push）
  - 🟡 **落后 N** —— 远程比本地多 N 个提交（可以 pull）
  - ⚪️ **失效** —— 目录被手动删了，但 git 里还留着引用，需要清理

任务标题旁还有个数字角标（这个任务有几个 worktree），如果组里有失效的 worktree，标题会出现一个 ⚠️ 提醒你清理。

### 能做什么

**创建（右上角「创建 Worktree」）** —— 这是最常用的入口：

1. 填**任务名**（建议直接用需求号，如 `PROJ-1234-需求简述`）——它会成为目录名
2. **勾选多个项目**（这个需求要改哪几个仓库就勾哪几个）
3. 填**分支名**（如 `feat/PROJ-1234`）
4. 默认勾选"创建新分支"；如果分支已存在、想检出已有分支，就取消勾选
5. 弹窗下方有**实时路径预览**，告诉你 worktree 会建在哪、用什么分支

点「创建」，工具会**为每个勾选的项目各建一个 worktree**，统一放进 `worktree根目录/任务名/项目名/`。某个项目失败不影响其他的，结束后会告诉你成功几个、失败几个（失败的会列出名字）。

**打开** —— 每个 worktree 行上有两个按钮：用 VSCode 打开、用 Finder 打开。任务组标题那行还能直接打开**整个任务目录**。

**删除** —— 删单个 worktree 时会**二次确认**。如果这个 worktree 有未提交的改动，工具会**先拒绝**并弹窗问你"确定要强制删除、丢弃这些改动吗"，避免误删没保存的工作。

**清理失效** —— 对标了"失效"的 worktree，点「清理」就会执行 `git worktree prune`，把 git 里的废引用清掉。

## 六、项目视图：管所有仓库的"主体"状态

切到「项目」视图，看到的是**源项目根目录下所有 git 仓库**的列表（注意：这里看的是仓库主工作区的状态，不是 worktree）。

顶部四个统计卡片一眼看清全局：**项目总数 / 非主分支 / 有未提交变更 / 可拉取更新**。

下面是项目表格，支持：

- **筛选**：全部 / 非主分支 / 有变更 / 可拉取
- **搜索**：按项目名
- **单项操作**：切回主分支、拉取更新、用 Finder / VSCode 打开
- **批量操作**：勾选多个项目后，一键执行：
  - 批量切到主分支
  - 批量拉取更新
  - 批量暂存变更（`git stash`）
  - 批量操作带进度条，结束告诉你成功/失败数

> 想看准确的"落后远程几个提交"，点右上角**「刷新」**（会带 fetch），或在设置里开启"自动 fetch"。不 fetch 的话，behind 数是基于上次同步的本地信息。

## 七、几个典型工作流

### 工作流 1：开新需求，一把建好所有 worktree

> 你接到 `PROJ-3000`，要改前端 PC、管理端小程序两个仓库。

1. 打开 Visual Worktree（默认就在 Worktree 视图）
2. 右上角「创建 Worktree」
3. 任务名填 `PROJ-3000-xxx`，勾选 `web-app` 和 `admin-wxapp`，分支填 `feat/PROJ-3000`，保持"创建新分支"勾选
4. 点「创建」—— 两个仓库的 worktree 一次建好，整齐地放在 `worktrees/PROJ-3000-xxx/` 下
5. 在新出现的任务组里，点 VSCode 按钮直接进对应 worktree 开干

**对比手动：** 省去了在每个仓库 `cd` 进去、敲 `git worktree add -b ...`、记路径的重复操作。

### 工作流 2：早上回来，扫一眼所有任务进度

1. 打开应用，Worktree 视图一览所有任务
2. 哪个 worktree 标了"🔴 有变更"——昨天没提交完的
3. 哪个标了"🟡 落后 N"——需要 pull 一下再继续
4. 哪个标了"⚪️ 失效"——之前删过目录，顺手点「清理」

**对比手动：** 不用一个个 `cd` 进去敲 `git status`。

### 工作流 3：需求上线了，收尾清理

> `PROJ-2001` 已合并上线，相关 worktree 可以删了。

1. Worktree 视图找到 `PROJ-2001` 任务组
2. 逐个点 worktree 的删除按钮
3. 如果某个 worktree 还有没提交的临时改动，工具会拦你一下，确认无误再强制删
4. 删干净后，如果还有失效引用，点「清理」收尾

### 工作流 4：批量把所有仓库切回主分支、拉最新

> 准备开新需求前，想让所有仓库回到干净的最新主分支。

1. 切到「项目」视图
2. 点「刷新」（带 fetch），看清谁落后了
3. 全选项目 → 批量操作 →「批量切到 master」
4. 再来一次「批量拉取更新」

## 八、常见问题

**Q：我在命令行手动建的 worktree，工具能识别吗？**
能。它读的是 `git worktree list` 的真实数据，只要 worktree 路径落在你配置的"worktree 根目录"下、且符合 `任务名/项目名` 结构，就会被归类显示。

**Q：为什么有的 worktree 显示"落后 N"不准 / 不显示？**
"落后远程"要先 `git fetch` 拿到远程最新引用才能算准。需要注意：**Worktree 视图自己的「刷新」不会 fetch**（只重新读本地状态）。要让落后数变准，去**「项目」视图点「刷新」**（它会带 fetch），或在设置里开启"自动 fetch"——因为 worktree 和主仓库共享同一份 `.git`，主仓库 fetch 完，回到 Worktree 视图刷新一下，落后数就更新了。

**Q：删 worktree 会不会误删我没提交的代码？**
不会无脑删。有未提交改动时工具会先拒绝，明确弹窗问你是否"强制删除、丢弃改动"，你确认了才删。

**Q："失效"是什么意思？**
你手动删了 worktree 的目录，但 git 内部还记着这个 worktree 的引用。这种状态标"失效"，点「清理」执行 `git worktree prune` 即可。

**Q：配置存在哪？想重置怎么办？**
存在 `~/.visual-worktree/config.json`。删掉它重开应用会恢复默认。
